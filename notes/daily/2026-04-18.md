## 踩坑 & 解决

### 坑 1：IsaacLab `main` 分支 ≠ Isaac Sim 4.5 可用版本 ⚠️

最关键的坑，差点直接 `./isaaclab.sh --install` 就炸。

**现状**：clone 下来默认在 `main`，最新 commit 已是 **v3.0.0-beta**。v3.0.0 用 **Newton 物理引擎**（不是 PhysX），要求 **Isaac Sim 5.x**。本地装的是 Isaac Sim 4.5.0 → 跟 `main` 不兼容。

**版本对应**：

| IsaacLab | Isaac Sim | 物理 |
|---|---|---|
| v2.x（v2.3.2 最新稳定） | 4.2 / 4.5 | PhysX |
| v3.0.0-beta（main） | 5.x | Newton |

**正确操作**：

```bash
cd ~/embodied-ai/isaaclab/IsaacLab
git checkout v2.3.2
./isaaclab.sh --install
```

**教训**：clone GitHub 上的框架项目，**永远先看 tag**，不要默认 `main`。Isaac Sim / IsaacLab / LeRobot 这种快速迭代的项目，`main` 经常是下一个大版本的开发分支。`git describe --tags` 快速确认当前 checkout 在哪个版本窗口。

### 坑 2：CUDA 999 错误（`CUDA_ERROR_UNKNOWN`）

PyTorch 装完第一次跑 `torch.cuda.is_available()` 报 CUDA error 999。

**原因**：驱动进入僵尸状态（可能休眠唤醒 / 扩展加载残留）。`nvidia-smi` 本身正常，但 CUDA runtime 初始化失败。

**解决**：**重启系统**，再跑就通。

**教训**：CUDA 999 = "unknown error"，几乎都是驱动/内核模块状态问题，不要花时间 debug `LD_LIBRARY_PATH` 或重装 PyTorch，**先重启**。仍报 999 再查 `dmesg | grep -i nvidia` 和 `nvidia-smi` 的 Persistence Mode。

---

### 坑 3：Shader 编译耗时 15 分钟 + GNOME 误报"无响应"

Isaac Sim 首次启动 GUI 时窗口长时间空白，GNOME 弹窗提示"应用程序无响应，是否强制关闭"。

**原因**：Isaac Sim 第一次启动编译 Omniverse shader 缓存到 `~/.cache/ov/`，4070 Ti Super 实测 **10-20 分钟**。主线程被占，窗口不刷新 → GNOME 判定卡死。

**解决**：**忍住不点"强制关闭"**，让它跑完。观察 `du -sh ~/.cache/ov/` 大小在涨就是还在工作。第二次启动只需 10-20 秒。

**教训**：首次启动慢是**预期行为**，不是 bug。换机器 / 清 cache 后要预留 20 分钟窗口。

---

### 坑 4：Isaac Sim 启动报 ROS2 Bridge startup failed

```
[Error] [isaacsim.ros2.bridge.scripts.extension] ROS2 Bridge startup failed
[ext: isaacsim.ros2.bridge-4.1.15] shutdown
```

**原因**：Isaac Sim 4.5 默认加载 ROS 2 Bridge，需要 Humble 的 `rclcpp` / FastDDS。但系统 `ROS_DISTRO` 为空，`/opt/ros` 不存在（ROS 2 按 plan 是 Week 3 Day 15 才装）。

**结论**：**非阻塞**。Week 1-2 的 Franka RL 训练不用 ROS 2 Bridge，Week 3 装完 Humble 自消。

**处理**：
- 方案 A（推荐）：忽略，继续主线
- 方案 B（想日志干净）：`isaacsim --/exts/isaacsim.ros2.bridge/enabled=false`
